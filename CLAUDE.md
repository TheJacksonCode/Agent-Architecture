# CLAUDE.md - Agent Architecture Designer

## Workflow standard: research + plans per version (since v32.8)

Kazda nietrywialna wersja (major visual overhaul, nowa grupa funkcji, redesign, audyt) musi miec wlasny folder `v{N}/` z podkatalogami `research/` i `plans/`. Ta struktura zostaje na stale w repo.

**Struktura obowiazkowa:**
- `v{N}/MASTER_PLAN.md` - plan fazowy (A -> I), constrainty, workflow
- `v{N}/PROGRESS.md` - log postepu aktualizowany po kazdej fazie (source of truth po kompakcji)
- `v{N}/research/R{1..N}_{topic}.md` - raporty researcherow (pixel measurements + drop-in CSS + sources)
- `v{N}/research/CRITIC.md` - walidacja researchu (PASS/REVISE per researcher, consensus, konflikty)
- `v{N}/plans/MANIFEST.md` - design decisions (DD01..DDN) z uzasadnieniem
- `v{N}/plans/DESIGN_SPEC.md` - konkretne tokeny CSS, recipes, utilities
- `v{N}/plans/LAYOUT_SPEC.md` - ASCII/HTML mockups, class names, grid
- `v{N}/plans/HITL_GATE_{1,2,3}.md` - bramy decyzyjne z opcjami dla uzytkownika
- `v{N}/plans/QA_H{1,2,3}_{topic}.md` + `QA_REPORT.md` - audyt koncowy

**Dlaczego to jest obowiazkowe:**
1. **Reuse researchu**: kolejna wersja (v32.9, v33...) czyta istniejace raporty R* z poprzednich wersji zanim zamowi nowe. Jesli temat juz byl pokryty (np. APCA contrast, glassmorphism, motion easing), NIE powtarzaj researchu - cytuj poprzedni plik i doloz tylko delta research na to czego brakowalo. Redukuje zuzycie tokenow i czas.
2. **Compaction recovery**: po kazdej kompakcji czytam MASTER_PLAN.md + PROGRESS.md i wracam do przerwanego miejsca bez zgadywania.
3. **Auditability**: kazda decyzja designerska ma papierowy slad z uzasadnieniem i zrodlem.
4. **HITL gates**: trzy bramy (Gate #1 przed Build, Gate #2 przed QA, Gate #3 przed ship) zapisuja opcje jakie uzytkownik widzial i co wybral.

**Zasada reuse (stala):** przed Phase B w nowej wersji ZAWSZE zrobic grep po istniejacych `v*/research/*.md` pod katem slow kluczowych tematu. Jesli znajde >=70% pokrycia tematu w istniejacym raporcie, delegowac do researchera tylko "delta" (co sie zmienilo od tamtego momentu, co nowe w docsach, co bylo luka). To obowiazuje dla tego projektu na stale.
