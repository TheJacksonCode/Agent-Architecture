# v32.16 EN Translation Style Guide

## Dialect
US English. Use: optimize/color/behavior/analyze/organize/favor/canceled/traveled. Avoid: optimise/colour/behaviour/analyse/organise/favour/cancelled/travelled.

## Punctuation
- **NO em-dashes** (`—`), **NO en-dashes** (`–`) - use regular hyphens `-` only
- Apostrophes allowed: `don't`, `it's`, `we'll`
- In JS strings, escape apostrophes with backslash: `'don\'t'`, `'it\'s'`
- Quotes inside strings: use smart single quotes escaped or just straight single quotes
- Ellipsis: use three dots `...` not `…`

## Tone
- Technical, direct, confident
- Active voice: "The agent generates..." not "reports are generated by..."
- Concrete over abstract: "runs 3 searches per minute" not "performs research activities"
- No corporate fluff: avoid "leverage", "synergies", "deliver value", "best-in-class", "world-class"
- Match the Polish original's register - if PL is punchy, EN is punchy; if PL explains, EN explains
- Keep the "top of the top" feeling - strong verbs, specific nouns, no weasel words

## Structural rules (ABSOLUTE - DO NOT VIOLATE)

1. **Keys stay in Polish** - all object keys like `tagline:`, `missionShort:`, `whoIs:`, `analogy:`, `howItWorks:`, `inputs:`, `outputs:`, `does:`, `doesNotDo:`, `antiPatterns:`, `keyConcepts:`, `stats:`, `bestFor:`, `worstFor:`, `relatedAgents:`, `relatedPresets:`, `glossary:`, `learningQuote:`, `realExample:` - these are code identifiers, NOT translated.

2. **IDs stay as-is** - `relatedAgents: ['orchestrator','synthesizer']` - those ARE IDs, never translated.

3. **Preserve counts** - if PL `does` has 7 items, EN `does` has 7 items. If PL `howItWorks` has 4 phases, EN has 4 phases. If PL `keyConcepts` has 5 terms, EN has 5 terms.

4. **Preserve types** - object stays object, array stays array, string stays string.

5. **Preserve field order** - keep the same field order as PL. Do not reorder.

6. **Label fields inside objects** (howItWorks phases, keyConcepts, glossary, stats):
   - `label:` and `term:` fields ARE translated to EN
   - `desc:` and `def:` and `definition:` fields ARE translated to EN
   - `value:` in stats is usually a number or short string - translate if it's a label like "ok 15 min", keep if it's a pure number

7. **No diacritics in Polish-origin names** - keep agent/preset technical names as they are (they stay in Polish source keys). But EN content can and should use normal English with all natural characters.

## Field-specific guidance

- **tagline**: one-line essence, under 80 chars. Keep punchy.
- **missionShort**: 2-3 sentences. What the thing delivers and for whom.
- **whoIs**: 2-3 sentences. Target user/situation.
- **analogy**: single metaphor sentence starting with "This agent is like..." / "This preset is like..." (match PL opener).
- **howItWorks**: array of 4-6 phase objects with `label` (phase name) and `desc` (what happens).
- **inputs / outputs**: arrays of concrete nouns. Keep them concrete and tangible.
- **does / doesNotDo**: arrays of 7-8 capabilities/limits. Keep each item concrete and scannable.
- **antiPatterns**: array of 5 "Name - description" entries. Keep the Name crisp.
- **keyConcepts**: array of 5 `{term, def}` objects. 1-sentence defs.
- **stats**: array of `{label, value}` objects. Translate labels, usually keep numeric values.
- **bestFor / worstFor**: arrays of "When..." / "When NOT..." items (3 each).
- **relatedAgents / relatedPresets**: IDs - DO NOT TRANSLATE.
- **glossary**: array of 5 `{term, definition}` objects. 1-sentence definitions.
- **learningQuote**: single inspirational/philosophical sentence.
- **realExample**: one-paragraph scenario starting with "Imagine that..." (match PL "Wyobraz sobie ze...").

## Output format

Each batch writes to its assigned file. Structure:

```javascript
// Batch: {name} EN translations
  orchestrator: { ...18 fields translated to EN... },
  synthesizer: { ... },
  ...
```

- NO wrapper `const = {` - just the body entries with trailing commas
- Match indent of source (2 spaces for top-level, 4 for nested)
- Preserve blank lines between entries if PL source has them

## Verification before submit

1. Count entries - must match input
2. Count fields per entry - must be 18 (or whatever the PL source had)
3. Grep for em-dashes (`—` or `–`) - must be ZERO
4. Grep for `'` inside strings - must be escaped `\\'`
5. Run mental `node --check` - object literal must be syntactically valid JS
