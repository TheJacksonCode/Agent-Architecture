SEKCJA 01

Dlaczego zespoly agentow?

Pojedynczy agent to jak one-man-band - moze grac na 6 instrumentach naraz, ale nigdy nie zagra jak orkiestra. 92% skutecznosci na zlozonych zadaniach vs 35% dla pojedynczego agenta.

Pojedynczy agent

Ograniczone okno kontekstowe. Przeskakuje miedzy rolami. Traci kontekst.
Przy 5 krokach z 85% accuracy: 44% sukcesu.
Przy 10 krokach: tylko 20%.

Zespol agentow

Kazdy agent = specjalista. Wlasny kontekst. Rownolegle przetwarzanie.
57% organizacji ma agentow w produkcji.
Rynek: $8.5 miliarda w 2026 (Deloitte).

P(sukces) = 0.85n
Kazdy krok z 85% accuracy zmniejsza calkowite szanse sukcesu wykladniczo
3 kroki = 61%
5 krokow = 44%
10 krokow = 20%
Dlaczego system wieloagentowy ma 92% skutecznosci vs 35% u pojedynczego agenta na zlozonych zadaniach?
Wielu agentow = wiecej mocy obliczeniowej
Kazdy agent specjalizuje sie w jednej domenie, ma waski kontekst i mniejsze ryzyko halucynacji
Systemy wieloagentowe uzywaja lepszych modeli AI
Wielu agentow moze pisac dluzsze odpowiedzi
💡 Did you know?

Pierwszy wieloagentowy system AI (DVMT - Distributed Vehicle Monitoring Testbed) powstal juz w 1980 roku na MIT. Potrzebowal 46 lat, zeby technologia dojrzala do produkcji na skale enterprise. Dzis 57% organizacji ma agentow w produkcji.

    📝 Shared Scratchpad Pattern

    KONCEPT 2026

    Wzorzec komunikacji, w ktorym agenty NIE rozmawiaja bezposrednio ze soba, lecz czytaja i pisza do wspolnej tablicy (scratchpad). Eliminuje problem N*N polaczen w teamie N agentow.

    Mechanizm: Centralny dokument JSON/Markdown z sekcjami per agent. Kazdy agent czyta calosc, ale modyfikuje TYLKO swoja sekcje. Orkiestrator monitoruje zmiany i triggeruje kolejne agenty.

    Zalety: Pelna transparentnosc (kazdy widzi co pisza inni), audit trail za darmo, zero zaginionych wiadomosci.
    Wady: Rosnacy kontekst - wymaga pruning strategii lub sliding window.

    Realna implementacja: MANIFEST.json w Observatory HIVE-MIND, shared context w Cursor Agent Mode, Claude Projects jako scratchpad.

    Twoj agent Claude musi odczytac dane z bazy PostgreSQL. Ktory protokol to umozliwia?
    A2A - agent z agenta
    MCP - agent do narzedzia (baza danych)
    ACP - standardowy format wiadomosci
    AG-UI - wyswietlenie danych
    💡 Did you know?

    Protokol MCP osiagnal 97 milionow pobrari SDK w pierwszym roku od release'u - szybciej niz Docker (72M) i Kubernetes (58M) w analogicznym okresie. Anthropic stworzyl de facto "USB-C dla AI" i kazdy vendor go zaadoptowal.

    ⚡ Wolny Elektron (Free Electron)

    Agent mostkujacy WSZYSTKIE fazy z bidirectional awareness. Mediator pattern odkryty intuicyjnie.

    Problem: Orkiestrator jest overloaded - nie moze sledzic kazdego detalu.
    Rozwiazanie: Niezalezny agent z dostepem do calego kontekstu, mostkujacy research <-> implementacja.
    Analogia: CTO w firmie - nie koduje, nie bada, ale wie co sie dzieje wszedzie i laczy kropki.

    👁 Narrow Context Principle

    Im glebiej w hierarchii, tym wezszy kontekst. Nie chodzi o oszczednosc - chodzi o jakosc.

    Level 0 (Orkiestrator): Pelny kontekst projektu
    Level 1 (Analityk): Cel + wymagania
    Level 2 (Researcher): Tylko swoja domena badawcza
    Level 3 (Koder): Tylko specyfikacja kodu do napisania
    Level 4 (QA): Tylko artefakt do sprawdzenia + kryteria

    🧬 Controlled Self-Organization

    KONCEPT 2026

    Hybrydowy wzorzec laczacy top-down orkiestracje z bottom-up emergencja. Orkiestrator definiuje CEL i OGRANICZENIA, ale NIE dyktuje dokladnych krokow.

    Zasady:
    1. Orkiestrator wyznacza mission boundary (cel + budzet + deadline + quality gate)
    2. Agenty SAME decyduja o kolejnosci zadan i podziale pracy
    3. Jesli agent utknie - moze poprosic innego agenta o pomoc (peer-to-peer)
    4. Orkiestrator interweniuje TYLKO gdy: przekroczony budzet, deadline, lub quality gate failed

    Analogia: Zespol Spotify (squad model) - autonomia w ramach misji. Nie micromanagement, ale tez nie chaos.

    Kiedy uzywac: Kreatywne zadania (design, content), eksploracja (research), prototypowanie.
    Kiedy NIE uzywac: Compliance, finanse, systemy safety-critical - tam potrzebujesz pelnej kontroli.

    Jaka jest glowna wartosc Devil's Advocate w Five Minds Protocol?
    Spowalnia prace zespolu, zeby byl bardziej ostrozny
    Zapobiega groupthink - kwestionuje kazda decyzje, co prowadzi do lepszych rozwiazan
    Eliminuje slabe pomysly na samym poczatku
    Zastepuje QA - nie trzeba testowac, bo Devil's Advocate juz sprawdzil
    SEKCJA 08

    9 krytycznych bledow

    Kazdy blad pochodzi z realnych wdrozen. 42% failow to bledy specyfikacji. 37% to breakdown koordynacji. Poznaj je zanim Cie spotkaja.

    ANTI-01

    🔁 Infinite Loop

    Agenty w petli krytyki bez limitu iteracji. "Popraw to" -> "Nie, wrocmy" -> ...

    Fix: max 3 cykle rewizji, potem decyzja orkiestratora
    ANTI-02

    👻 Hallucination Cascade

    Agent A halucynuje -> Agent B buduje na fałszywych danych -> system "pewnie" klamie

    Fix: source tracking + walidacja miedzyagentowa
    ANTI-03

    💸 Token Waste

    15 agentow do zadania na 3. Over-engineering z ceremonia zamiast kodu.

    Fix: zacznij od 3 agentow, skaluj gdy potrzebujesz
    ANTI-04

    🚧 Over-Engineering

    800-liniowy katalog agentow zamiast 80 linii kodu + RUN. Planowanie > budowanie.

    Fix: buduj JEDEN agent e2e zanim zaprojektujesz 14
    ANTI-05

    💥 Context Pollution

    Za duzo kontekstu = gorsza jakosc. Agent dostaje caly repo zamiast jednego pliku.

    Fix: Narrow Context Principle - tylko to co niezbedne
    ANTI-06

    🔥 Cascading Failures

    Jeden agent pada -> caly system pada. Brak graceful degradation.

    Fix: circuit breaker pattern + fallback strategies
    ANTI-07

    👨‍👨‍👦‍👦 Coordination Overhead

    Diminishing returns po 11+ agentach. Wiecej czasu na koordynacje niz na prace.

    Fix: 6-10 agentow = sweet spot dla wiekszosci zadan
    ANTI-08

    🙌 Sycophancy Chain

    Agenty zgadzaja sie ze soba zamiast kwestionowac. "Swietny pomysl!" x 5 agentow.

    Fix: Devil's Advocate + adversarial review
    ANTI-09

    🔄 Schema Drift

    Dane mutuja miedzy agentami jak w gluchym telefonie. JSON IN != JSON OUT.

    Fix: JSON Schema validation miedzy agentami
    Agent A halucynuje fakt, Agent B bierze go jako prawde i buduje na nim analize, Agent C cytuje analize jako "zweryfikowane zrodlo". Jaki to anti-pattern?
    Infinite Loop - agenty sie zapetlily
    Hallucination Cascade - halucynacja propaguje sie przez system
    Sycophancy Chain - agenty sie ze soba zgadzaja
    Schema Drift - dane sie zmienily

    TOKENY I MODELE

    Tokeny, modele i koszty

    Zanim zaprojektujesz system agentowy, musisz rozumiec tokeny - walute AI. Kazde wywolanie modelu kosztuje tokeny. Roznica miedzy modelami to nawet 190x.

    🔢 Co to sa tokeny?

    Token to kawalek tekstu - nie zawsze cale slowo. Modele AI nie widza liter - widza tokeny. Tokenizacja (BPE - Byte Pair Encoding) dzieli tekst na czestotliwe fragmenty.

    Srednia: 1 token ≈ 4 znaki (angielski) / ~3 znaki (polski).
    1 strona tekstu ≈ 500-600 tokenow. 200K tokenow ��� 500 stron ≈ cala ksiazka.

    Przyklad: "Czesc, jak sie masz?" → ["Cze", "sc", ",", " jak", " sie", " ma", "sz", "?"] = 8 tokenow

    Dlaczego to wazne: Platisz za KAZDY token - input AND output osobno. Opus input: $5/1M tokenow. Output: $25/1M. Wiec 50K input + 2K output = $0.25 + $0.05 = $0.30/run.

    💡 Context Window - dlaczego wieksze nie znaczy lepsze

    200K tokenow (Claude) = ~150K slow = ~500 stron. 1M tokenow (Gemini) = ~2500 stron.

    Ale: wieksz kontekst = wyzszy koszt + wieksza latencja + attention degradation.
    LLM "gubi sie" w dlugich kontekstach (Lost in the Middle problem). Dlatego Narrow Context Principle jest tak wazny - dawaj agentowi TYLKO to co potrzebuje.

    Twoj agent musi przeanalizowac 100-stronicowy raport prawny i wygenerowac 2-stronicowe podsumowanie. Ile tokenow input/output potrzebuje?

    Input: ~50K, Output: ~1.2K - 100 stron × ~500 tok/strone, 2 strony output
    Input: ~500K, Output: ~50K - zdecydowanie za duzo, marnowanie tokenow
    Input: ~1K, Output: ~10K - odwrocone proporcje

    MULTI-AGENT
    Jak zaprojektowalbys multi-agent workflow dla systemu RAG?
    Dekompozycja na 4 agentow: 1) Ingestion Agent (chunking, embedding) → 2) Retrieval Agent (semantic search + BM25 hybrid, reranking) → 3) Generation Agent (LLM z kontekstem) → 4) Validation Agent (hallucination detection, source citation).

    LangGraph dla stateful orchestration. Quality gates miedzy etapami. 80% failow RAG to chunking/ingestion, nie LLM - wiec inwestuj w Ingestion Agent. Agentic RAG z retrieve-evaluate-revise loops > naive single-pass.
    MULTI-PROVIDER
    Jak zarzadzasz wieloma dostawcami modeli w jednym systemie?
    Warstwa abstrakcji (LiteLLM lub custom router). Fallback chain: Opus → Sonnet → GPT-4o. Model routing based on task complexity. Unified API interface. Circuit breaker per provider. Cost tracking per request.

    Klucz: 190x roznica kosztowa miedzy frontier (Opus $25/1M output) a small (Haiku $4/1M). Routing to najwazniejsza optymalizacja kosztowa - 40-60% redukcji.
    SECURITY
    Jak zapewniasz security w systemie agentowym?
    6 zasad: Least Privilege, HITL, Sandboxing, Inter-agent Validation, Audit Logging, Rate Limiting.
    Plus: prompt injection detection (input/output filtering), data classification, mTLS between agents.

    Kluczowy insight: 88% organizacji mialo incydenty security z agentami (2026). Governance-containment gap to #1 problem - organizacje moga monitorowac ale nie moga ZATRZYMAC agentow. NIST wydal RFI o bezpieczenstwie agentow w styczniu 2026.
